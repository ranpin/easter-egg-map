## 1. 产品概述

### 1.1 产品定位

**彩蛋地图**是一款基于地理位置的 UGC 内容聚合小程序/H5 应用。用户在旅行途中发现或放置的"彩蛋"（如藏在景点的教科书、国家地理杂志、手写信件、盲盒等），可以通过本平台在地图上标记、记录和分享，形成一个全球范围的"旅行彩蛋寻宝网络"。

### 1.2 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| 🎯 寻宝者 | 热爱旅行、喜欢探索隐藏惊喜的年轻人（18-35岁） | 在旅途中发现有趣的彩蛋点位，获得探索乐趣 |
| 🎁 藏宝者 | 喜欢在旅途中留下物品、传递温暖的旅行者 | 记录自己放置的彩蛋，看到他人发现的反馈 |
| 📱 内容消费者 | 对旅行文化感兴趣但不一定出行的用户 | 浏览各地彩蛋故事，获取旅行灵感 |
| 🏪 商家/文旅机构 | 景区、咖啡馆、书店、民宿等 | 通过彩蛋活动引流，提升品牌曝光 |

### 1.3 核心价值主张

- **对C端用户**：让每一次旅行都变成一场寻宝游戏，发现陌生人留下的温暖与惊喜
- **对B端合作伙伴**：将线下空间融入彩蛋网络，用低成本创造高传播的文旅体验

## 2. 市场分析

### 2.1 市场背景

- 2025年国内旅游人次突破60亿，Z世代成为主力消费群体
- "City Walk""宝藏小店""隐藏打卡点"等概念在小红书、抖音持续火爆
- Geocaching（地理藏宝）在全球拥有超300万活跃用户，验证了"地理+寻宝"模式的可行性
- 国内尚无专注于"旅行彩蛋"的垂直平台，存在市场空白

### 2.2 竞品分析

| 竞品 | 类型 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------|-------------|
| Geocaching | 全球地理藏宝 | 成熟社区、全球覆盖 | 界面老旧、中文生态弱、偏硬核户外 | 更轻量、更文艺、强社交属性 |
| 小红书/抖音 | 内容平台 | 流量大、内容丰富 | 信息分散、无地图聚合、无法结构化 | 专注彩蛋垂类、地图可视化、结构化数据 |
| 高德/百度地图 | 地图工具 | 地图能力强 | 非UGC内容平台、无彩蛋概念 | 以彩蛋为核心的社区体验 |

### 2.3 SWOT 分析

- **Strengths**: 独特定位、情感共鸣强、低门槛参与
- **Weaknesses**: 冷启动内容不足、UGC质量管控难
- **Opportunities**: 文旅复苏、City Walk热潮、品牌联名需求旺盛
- **Threats**: 大厂可能复制、线下彩蛋真实性验证困难

## 4. 技术方案

### 4.1 技术选型

| 层级 | 技术选择 | 理由 |
|------|---------|------|
| 前端框架 | React + TypeScript | 生态成熟、组件丰富、类型安全 |
| UI 组件库 | Tailwind CSS + shadcn/ui | 高度可定制、设计感强、轻量 |
| 地图服务 | **高德地图 JS API 2.0**（主）+ Mapbox GL JS（海外备用） | 高德国内POI数据最完整，景点/商圈/地标信息丰富，支持卫星图/3D建筑/室内地图；放大到街道级别自动显示周边景点POI；免费额度充足（日均30万次） |
| 后端服务 | Node.js (Express/Fastify) + PostgreSQL | 轻量高效、PostGIS 支持地理查询 |
| ORM | Prisma | 类型安全、迁移管理方便 |
| 对象存储 | 阿里云 OSS / AWS S3 | 图片存储 |
| 部署 | Vercel (前端) + Railway/Fly.io (后端) | 快速部署、自动CI/CD |
| 小程序 | Taro / uni-app（可选） | 一套代码多端发布 |
| 数据采集 | Python (requests/playwright) + 定时任务 | 小红书/抖音彩蛋内容采集，详见4.5节 |

#### 4.1.1 地图API选型说明（v1.1更新）

**为什么从高德替代Mapbox作为国内主地图：**

| 对比维度 | 高德地图 JS API 2.0 | Mapbox GL JS |
|---------|-------------------|-------------|
| 国内POI数据 | ✅ 极其丰富，景点/商圈/地标全覆盖 | ❌ 国内数据薄弱，很多地点无标注 |
| 缩放体验 | ✅ 逐级加载：国家→省→市→区→街道→POI，放大自动显示景点名称与图标 | ⚠️ 中文POI稀疏，放大后大量空白 |
| 卫星影像 | ✅ 高清国内卫星图 | ⚠️ 国内分辨率低 |
| 合规性 | ✅ 符合国内地图法规，GCJ-02坐标系 | ❌ WGS-84坐标需偏移，存在合规风险 |
| 免费额度 | ✅ 个人开发者日均30万次 | ⚠️ 免费版月5万次 |
| 自定义标记 | ✅ 支持自定义Marker/InfoWindow/海量点 | ✅ 同样支持 |
| 海外数据 | ⚠️ 一般 | ✅ 全球数据优秀 |

**双源策略**：国内用户默认高德地图；检测到海外IP或用户手动切换时启用Mapbox。MVP阶段先实现高德地图。

## 5. 界面设计要求

### 5.1 设计原则（v1.1更新）

- **小程序风格**：采用移动端优先的小程序交互范式——底部Tab导航栏、顶部状态栏留白、卡片式内容布局、下拉刷新/上拉加载、页面转场动画
- **审美感优先**：参考小红书/马蜂窝的视觉水准，注重排版节奏、字体层级、色彩和谐度；避免"工具感"，追求"杂志感"
- **地图沉浸**：首页即全屏地图，UI元素以半透明浮层形式叠加，不遮挡地图内容；彩蛋标记使用自定义手绘风图标
- **情感化微交互**：发现彩蛋时的弹跳动画、收藏时的心形粒子效果、页面切换的丝滑过渡
- **响应式适配**：以375px移动端为基准设计，桌面端自动居中显示手机尺寸容器或自适应扩展

### 5.2 核心页面（v1.1更新）

1. **首页（地图页）**：全屏高德地图 + 底部悬浮搜索栏（圆角胶囊样式）+ 右上角分类筛选浮层 + 自定义彩蛋Marker（按分类着色）+ 点击Marker弹出底部半屏详情卡片（可上滑展开全屏）
2. **发现页**：瀑布流卡片列表，每张卡片包含封面图+标题+地点标签+来源平台icon；支持下拉刷新、上拉加载更多、分类Tab切换
3. **彩蛋详情页**：沉浸式大图轮播（支持手势滑动）+ 毛玻璃信息面板 + 地图缩略图定位 + 来源链接跳转 + 收藏/分享按钮 + 评论区
4. **发布页**：表单式布局，拍照/选图组件 + 自动定位（高德逆地理编码）+ 分类选择器 + 描述输入 + 来源链接
5. **我的页**：头像+昵称+成就徽章墙 + Tab切换（我发布的/我的收藏/我的足迹）+ 设置入口

### 5.3 设计规范（v1.1更新）

#### 配色方案
- 主色：温暖珊瑚橙 `#FF6B6B` — 探索与惊喜
- 辅色：薄荷绿 `#4ECDC4` — 自然与旅行
- 强调色：琥珀金 `#F4A261` — 彩蛋高亮/成就
- 背景：暖米白 `#FAFAF5` — 温暖底色
- 卡片背景：纯白 `#FFFFFF` + 柔和阴影 `0 2px 16px rgba(0,0,0,0.06)`
- 文字主色：墨灰 `#2D3436`
- 文字次色：雾灰 `#636E72`

#### 字体规范
- 标题：PingFang SC Medium / SF Pro Display Semibold，18-24px
- 正文：PingFang SC Regular / SF Pro Text，14-16px，行高1.6
- 辅助文字：12px，字间距0.5px

#### 圆角与间距
- 卡片圆角：16px
- 按钮圆角：24px（胶囊型）
- 页面内边距：16px
- 卡片间距：12px

#### 小程序风格交互规范
- 底部Tab栏高度：56px + safe-area-inset-bottom
- 顶部导航栏：透明/毛玻璃效果，高度44px + status-bar-height
- 弹窗/面板：从底部滑入，带橡皮筋回弹效果
- 列表项：左滑操作（收藏/分享）、长按预览
- 加载状态：骨架屏替代传统loading spinner
- 空状态：手绘风插画 + 引导文案

## 附录A：数据采集技术方案（v1.1新增）

### A.1 采集目标

从小红书、抖音等平台自动采集"旅行彩蛋"相关内容，结构化入库作为平台种子数据。

### A.2 采集关键词矩阵

| 平台 | 搜索关键词 | 预期内容类型 |
|------|-----------|-------------|
| 小红书 | 旅行彩蛋、景点藏宝、旅途惊喜、藏在景区的礼物、旅行盲盒、城市寻宝 | 图文笔记，含位置标签 |
| 抖音 | 旅行彩蛋、景区隐藏惊喜、在路上放了一本书、旅行宝藏、城市探险彩蛋 | 短视频，含定位信息 |
| 微博 | 旅行彩蛋、景点发现、旅途温暖 | 图文博文 |

### A.3 技术架构

```
┌─────────────┐    ┌──────────────┐    ┌─────────────┐    ┌────────────┐
│  定时调度器   │───→│  采集引擎     │───→│  数据清洗管道 │───→│  人工审核台 │
│ (Cron/APScheduler)│ (Playwright/API) │    │  (Python ETL) │    │  (Web后台)  │
└─────────────┘    └──────────────┘    └─────────────┘    └────────────┘
                                                │
                                         ┌──────▼──────┐
                                         │ PostgreSQL  │
                                         │ + PostGIS   │
                                         └─────────────┘
```

### A.4 采集策略

| 阶段 | 方式 | 频率 | 说明 |
|------|------|------|------|
| Phase 1 | 平台开放API + 关键词搜索 | 每日1次 | 合规优先，使用官方数据接口 |

### A.6 合规与风控

- **robots.txt遵守**：仅采集公开可见内容，不突破登录墙
- **频率控制**：单IP每分钟≤5次请求，避免对平台造成压力
- **内容标注**：所有采集内容明确标注来源平台和原作者
- **版权尊重**：提供"删除我的内容"入口，响应权利人下架请求
- **数据脱敏**：不采集用户隐私信息（手机号、真实姓名等）
