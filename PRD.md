## 1. 产品概述

### 1.1 产品定位

**彩蛋地图**是一款基于地理位置的 UGC 内容聚合小程序/H5 应用。用户在旅行途中发现或放置的"彩蛋"（如藏在景点的教科书、国家地理杂志、手写信件、盲盒等），可以通过本平台在地图上标记、记录和分享，形成一个全球范围的"旅行彩蛋寻宝网络"。

### 1.2 目标用户

| 用户角色 | 描述 | 核心需求 |
|---------|------|---------|
| 🎯 寻宝者 | 热爱旅行、喜欢探索隐藏惊喜的年轻人（18-35岁） | 在旅途中发现有趣的彩蛋点位，获得探索乐趣 |
| 🎁 藏宝者 | 喜欢在旅途中留下物品、传递温暖的旅行者 | 记录自己放置的彩蛋，看到他人发现的反馈 |
| 📱 内容消费者 | 对旅行文化感兴趣但不一定出行的用户 | 浏览各地彩蛋故事，获取旅行灵感 |
| 🏪 商家/文旅机构 | 景区、咖啡馆、书店、民宿等 | 通过彩蛋活动引流，提升品牌曝光 |

### 1.3 核心价值主张

- **对C端用户**：让每一次旅行都变成一场寻宝游戏，发现陌生人留下的温暖与惊喜
- **对B端合作伙伴**：将线下空间融入彩蛋网络，用低成本创造高传播的文旅体验

## 2. 市场分析

### 2.1 市场背景

- 2025年国内旅游人次突破60亿，Z世代成为主力消费群体
- "City Walk""宝藏小店""隐藏打卡点"等概念在小红书、抖音持续火爆
- Geocaching（地理藏宝）在全球拥有超300万活跃用户，验证了"地理+寻宝"模式的可行性
- 国内尚无专注于"旅行彩蛋"的垂直平台，存在市场空白

### 2.2 竞品分析

| 竞品 | 类型 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------|-------------|
| Geocaching | 全球地理藏宝 | 成熟社区、全球覆盖 | 界面老旧、中文生态弱、偏硬核户外 | 更轻量、更文艺、强社交属性 |
| 小红书/抖音 | 内容平台 | 流量大、内容丰富 | 信息分散、无地图聚合、无法结构化 | 专注彩蛋垂类、地图可视化、结构化数据 |
| 高德/百度地图 | 地图工具 | 地图能力强 | 非UGC内容平台、无彩蛋概念 | 以彩蛋为核心的社区体验 |

### 2.3 SWOT 分析

- **Strengths**: 独特定位、情感共鸣强、低门槛参与
- **Weaknesses**: 冷启动内容不足、UGC质量管控难
- **Opportunities**: 文旅复苏、City Walk热潮、品牌联名需求旺盛
- **Threats**: 大厂可能复制、线下彩蛋真实性验证困难

## 3. 功能架构

### 3.1 功能全景图

```
彩蛋地图
├── 🗺️ 地图探索模块
│   ├── 世界地图浏览（支持缩放、区域筛选）
│   ├── 彩蛋点位标记与聚类
│   ├── 附近彩蛋推荐
│   ├── 分类筛选（书籍/杂志/手写信/盲盒/其他）
│   └── 搜索（地点名/关键词/标签）
├── 📍 彩蛋详情模块
│   ├── 彩蛋卡片（图片+描述+位置+时间）
│   ├── 来源平台链接（抖音/小红书原文）
│   ├── 发现/未找到状态反馈
│   ├── 评论区
│   └── 收藏/分享
├── ✏️ 发布模块
│   ├── 新增彩蛋（拍照+定位+描述+分类）
│   ├── 引用外部平台内容（粘贴链接自动解析）
│   ├── 草稿箱
│   └── 发布审核提示
├── 👤 个人中心
│   ├── 我的发布 / 我的收藏 / 我的足迹
│   ├── 成就系统（发现N个彩蛋、覆盖N个城市）
│   ├── 个人主页（可分享）
│   └── 设置与反馈
└── 🏪 B端管理后台（v2.0）
    ├── 商家入驻与认证
    ├── 官方彩蛋活动管理
    ├── 数据看板
    └── 广告投放管理
```

### 3.2 MVP 功能范围（v1.0）

MVP 聚焦**地图探索 + 彩蛋详情 + 基础发布**三个核心模块，确保最小可用闭环：

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | 地图浏览与彩蛋标记 | 核心体验，必须首发 |
| P0 | 彩蛋详情页 | 展示彩蛋图文信息与来源 |
| P0 | 彩蛋列表/搜索 | 支持按城市、分类筛选 |
| P1 | 用户发布彩蛋 | 支持拍照+定位+描述 |
| P1 | 用户注册/登录 | 微信一键登录 |
| P1 | 收藏功能 | 收藏职位感兴趣的彩蛋 |
| P2 | 评论与互动 | 增强社区氛围 |
| P2 | 成就系统 | 提升留存与粘性 |
| P3 | B端后台 | v2.0 再考虑 |

## 4. 技术方案

### 4.1 技术选型

| 层级 | 技术选择 | 理由 |
|------|---------|------|
| 前端框架 | React + TypeScript | 生态成熟、组件丰富、类型安全 |
| UI 组件库 | Tailwind CSS + shadcn/ui | 高度可定制、设计感强、轻量 |
| 地图服务 | Mapbox GL JS / 高德地图 | Mapbox 国际化好；高德国内体验佳；可做双源适配 |
| 后端服务 | Node.js (Express/Fastify) + PostgreSQL | 轻量高效、PostGIS 支持地理查询 |
| ORM | Prisma | 类型安全、迁移管理方便 |
| 对象存储 | 阿里云 OSS / AWS S3 | 图片存储 |
| 部署 | Vercel (前端) + Railway/Fly.io (后端) | 快速部署、自动CI/CD |
| 小程序 | Taro / uni-app（可选） | 一套代码多端发布 |

### 4.2 数据模型（核心）

```
EasterEgg（彩蛋）
├── id: UUID
├── title: String          // 标题
├── description: Text      // 描述
├── category: Enum         // 书籍/杂志/手写信/盲盒/其他

├── city: String           // 城市
├── country: String        // 国家
├── images: String[]       // 图片URL列表
├── sourceUrl: String?     // 原始来源链接（抖音/小红书）
├── sourcePlatform: Enum?  // 来源平台
├── status: Enum           // 已确认/待验证/已移除
├── placedAt: DateTime?    // 放置时间
├── createdBy: UUID        // 发布者ID
├── createdAt: DateTime
└── updatedAt: DateTime

User（用户）
├── id: UUID
├── nickname: String
├── avatar: String
├── wechatOpenId: String?

Comment（评论）
├── id: UUID
├── content: Text

└── createdAt: DateTime

Favorite（收藏）
├── userId: UUID

└── createdAt: DateTime
```

## 5. 界面设计要求

### 5.1 设计原则

- **简洁精美**：大量留白、柔和配色、圆角卡片，避免信息过载
- **地图优先**：首页即地图，彩蛋以精美图标呈现，点击展开详情卡片
- **情感化设计**：温暖的插画风格图标、发现彩蛋时的微动效与音效
- **响应式**：完美适配手机、平板、桌面端

### 5.2 核心页面

1. **首页（地图页）**：全屏地图 + 底部搜索栏 + 浮动筛选按钮 + 彩蛋标记点
2. **彩蛋详情页**：沉浸式图片轮播 + 信息卡片 + 地图缩略图 + 评论区
3. **发布页**：表单式布局，支持拍照/选图、自动定位、分类选择
4. **个人中心**：头像+昵称+成就徽章+Tab切换（发布/收藏/足迹）
5. **列表页**：瀑布流/列表视图切换，支持排序与筛选

### 5.3 配色方案建议

- 主色：温暖的珊瑚橙 `#FF6B6B` — 代表探索与惊喜
- 辅色：深海蓝 `#4ECDC4` — 代表地图与旅行
- 背景：米白 `#FAFAF5` — 温暖不刺眼
- 文字：深灰 `#2D3436` — 清晰可读

## 6. 运营策略

### 6.1 冷启动计划

| 阶段 | 时间 | 目标 | 动作 |
|------|------|------|------|
| 种子期 | 第1-2月 | 50-100个精选彩蛋 | 人工整理抖音/小红书热门内容，聚焦成都、大理、西安、厦门、长沙 |
| 内测期 | 第3月 | 500注册用户 | 邀请旅行KOL体验，收集反馈 |
| 公测期 | 第4-5月 | 5000用户、500个彩蛋 | 开放UGC，举办"城市寻宝赛"活动 |
| 增长期 | 第6月起 | 持续增长 | 品牌合作、电商上线、跨城市联动 |

### 6.2 内容采集策略

- **Phase 1（人工）**：团队手动从抖音/小红书搜集高质量彩蛋内容，标注来源
- **Phase 2（半自动）**：开发爬虫辅助工具，自动抓取关键词内容，人工审核入库
- **Phase 3（UGC）**：开放用户投稿，建立审核机制（AI初审+人工复审）

## 8. 风险与应对

| 风险 | 等级 | 应对措施 |
|------|------|---------|
| 内容真实性难以验证 | 高 | 建立信用体系+社区举报+AI图像识别辅助审核 |
| 冷启动内容不足 | 高 | 前期投入人力精选，联合KOL共创种子内容 |
| 法律风险（放置物品合规性） | 中 | 明确用户协议、禁止危险/违法物品、与景区合作规范化 |
| 大厂复制 | 中 | 深耕社区文化与品牌调性，建立先发优势 |
| 地图服务成本 | 低 | 初期免费额度足够，增长后评估自建瓦片服务 |
